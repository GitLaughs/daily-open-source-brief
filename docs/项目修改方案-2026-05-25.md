# 项目修改方案

制定日期：2026-05-25

基于对全仓库代码、配置、文档和部署脚本的完整审阅，本文档列出所有值得实施的修改项，按优先级分批，每项包含背景、方案、影响范围和验收标准。

---

## 全局原则

- 不破坏现有授权方式（GITHUB_TOKEN、SMTP、LARK、cc-switch）。
- 单源失败不阻断日报主流程。
- 新增功能默认关闭或低风险启用。
- SQLite 继续作为唯一事实源，飞书只做入口和展示层。
- 配置驱动优先于硬编码。
- 新增处理能力优先进入插件阶段；需要 collector 后统一处理的能力先补 `enricher/postprocessor` 阶段，不在 `runner.py` 或 `run_daily.py` 里散落硬编码。
- 数据库中的 `items.score` 默认保存基础分；用户反馈、增量推送、临时过滤等运行期策略写入 `PluginContext.state` 的派生字段，不覆盖原始基础分。

## 第 0 批：管线基础修正（后续功能前置）

### 0.1 补齐 collector 后处理插件阶段

**背景**：截止日期抽取、反馈评分、跨源去重都发生在 collector 完成之后、summarizer 之前。若直接写在 `runner.py` 或内置 collector 里，会违反插件优先原则，也会让后续功能互相耦合。

**方案**：

新增插件阶段，推荐命名为 `enricher`（或 `postprocessor`，实现时二选一保持统一）：

```python
# app/runner.py
manager.run_stage("collector", ctx)
manager.run_stage("enricher", ctx)
manager.run_stage("summarizer", ctx)
```

典型插件：

- `feedback`: 根据用户反馈计算 `effective_score`，只影响排序和推送。
- `deadline`: 从条目标题、摘要、详情正文中抽取截止事项。
- `cross_source_dedupe`: 对 `ctx.state["web_items"]` 做跨源去重。

配置放入 `config/plugins.yml`：

```yaml
plugins:
  feedback:
    kind: enricher
    enabled: true
  deadline:
    kind: enricher
    enabled: true
  cross_source_dedupe:
    kind: enricher
    enabled: false
```

**影响文件**：

- 修改：`app/plugins/registry.py`（阶段顺序）
- 修改：`app/plugins/manager.py`（默认插件配置支持新 kind）
- 修改：`app/plugins/builtins.py`（注册内置 enricher 插件）
- 修改：`app/runner.py`（collector 后运行新阶段）
- 修改：`config/plugins.yml`
- 修改：`tests/test_plugins.py`

**验收标准**：

- `python -m app.cli plugin list` 能显示 `enricher` 插件。
- `plugin_runs` / `plugin_health` 记录新阶段插件状态。
- 禁用某个 enricher 不影响 collector、summarizer、renderer、sender 正常运行。

## 第 1 批：高优先级（直接提升日常体验）

### 1.1 截止日期抽取

**背景**：校园通知中"报名截止 5月30日""考试时间 6月10日"是用户最需要第一时间看到的信息，当前只按关键词加权，没有提取具体时间点。

**方案**：

新增 `app/deadline_extractor.py`：

```python
# 核心接口
def extract_deadlines(title: str, snippet: str, detail_text: str | None = None) -> list[DeadlineCandidate]:
    """从标题/摘要/详情页正文中抽取截止日期"""

@dataclass
class DeadlineCandidate:
    title: str          # 原始通知标题
    deadline: date      # 抽取的截止日期
    event_type: str     # 报名/申报/考试/答辩/确认/讲座
    confidence: float   # 0-1 置信度
    source_url: str     # 原始链接
    item_id: int | None # 关联 items 表 ID
```

新增 SQLite 表：

```sql
CREATE TABLE IF NOT EXISTS deadline_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  item_id INTEGER,
  title TEXT NOT NULL,
  event_type TEXT NOT NULL,
  deadline TEXT NOT NULL,
  confidence REAL NOT NULL DEFAULT 0,
  location TEXT,
  status TEXT NOT NULL DEFAULT 'pending',
  lark_task_id TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_deadline_events_deadline ON deadline_events(deadline);
CREATE INDEX IF NOT EXISTS idx_deadline_events_status ON deadline_events(status);
```

抽取规则（正则 + 启发式）：

- `报名截止[：:]?\s*(\d{4})[年.-](\d{1,2})[月.-](\d{1,2})`
- `截止时间[：:]?\s*(\d{4})[年.-](\d{1,2})[月.-](\d{1,2})`
- `考试时间[：:]?\s*(\d{4})[年.-](\d{1,2})[月.-](\d{1,2})`
- `答辩时间[：:]?\s*(\d{4})[年.-](\d{1,2})[月.-](\d{1,2})`
- 补充：`(\d{1,2})月(\d{1,2})日.*(截止|报名|考试|答辩)`

置信度规则：

- 标题中同时有事件词 + 完整日期：0.9
- 只在摘要中找到：0.7
- 只找到月日没有年份（推断当年）：0.5
- 日期已过：0.1（标记为 `expired`）

**影响文件**：

- 新增：`app/deadline_extractor.py`
- 修改：`app/db.py`（加表 + CRUD）
- 修改：`app/plugins/builtins.py`（新增 `DeadlineEnricherPlugin`，在 `enricher` 阶段运行）
- 修改：`config/plugins.yml`（新增 `deadline` 插件配置，默认启用或按需启用）
- 修改：`app/summarize.py`（读取 `ctx.state["deadline_events"]`，在"今日优先处理"置顶截止事项）
- 修改：`app/render.py`（读取 deadline 数据，HTML 中增加截止事项区域）
- 新增：`tests/test_deadline_extractor.py`
- 修改：`tests/test_plugins.py`（覆盖 deadline 插件注册和运行）

**验收标准**：

- 对教务处"选课确认截止 2026年6月5日"能正确抽取。
- 对"答辩时间：6月10日 上午8:30"能抽取日期。
- 已过期日期标记为 expired，不置顶。
- 截止事项在飞书和 HTML 日报中排在最前面。

### 1.4 飞书消息长度控制与摘要降级

**背景**：`lark_sender.py` 已通过 `split_markdown()` 按 `LARK_MAX_MARKDOWN_CHARS` 分块发送，但当 web + rss 条目多时，飞书会变成多条长消息，阅读体验差，也更容易触发发送失败。真正需要的是飞书专用精简摘要，而不是覆盖邮件/HTML 的完整日报。

**方案**：

## 第 2 批：中优先级（提升系统健壮性）

### 2.1 采集阶段并发

**背景**：当前 github、webpage、rss 三个 collector 串行执行。GitHub API 响应约 2-5s，webpage 每个源约 3-8s（共 4 个源），rss 每个源约 2-5s（共 3 个源）。总采集时间约 30-60s，通过并发可压缩到 10-15s。

**方案**：

不要共享 `ctx.conn` 或直接让多个 collector 同时修改同一个 `PluginContext.state`。SQLite WAL 支持多连接并发读写，但当前 `db.connect()` 返回的 sqlite 连接默认不能跨线程使用；同时 `ctx.state` 是普通 dict/list，不适合多线程直接写。

推荐改法：并发只覆盖网络抓取阶段，数据库写入和 `ctx.state` 合并保留在主线程。

修改 collector 插件结构：

```python
class WebpageCollectorPlugin(BasePlugin):
    def collect(self, ctx) -> dict:
        # 只做网络请求和解析，不写 DB，不写 ctx.state
        return {"web_items": web_items, "source_errors": web_errors, "source_runs": web_runs}

    def persist(self, ctx, data) -> PluginResult:
        # 主线程写 DB，合并 ctx.state
        ...
```

修改 `PluginManager`：

```python

```

若短期不想改插件接口，优先只在 `fetch_webpages_from_config()` 内部并发抓多个网页源、在 `fetch_rss_from_config()` 内部并发抓多个 RSS 源；这比跨插件并发风险更低。

**影响文件**：

- 修改：`app/plugins/manager.py`
- 修改：`app/plugins/builtins.py`（collector 拆分 `collect()` / `persist()`，或先在网页/RSS fetch 函数内部并发）

- 单个 collector 失败不影响其他 collector。
- 最终 `ctx.state` 中的 `item_ids`、`web_items`、`source_errors` 正确合并。

def parse_webpage_entries(text, page_url, source, *, today):
    soup = BeautifulSoup(text, "html.parser")
    candidates = []
    # 优先找 <ul>/<ol> 中的 <li>，其次找 <table> 中的 <tr>
    for li in soup.select("ul li, ol li"):

### 2.4 LLM 调用重试机制

**背景**：`summarize.py` 的 `summarize_with_llm()` 对 LLM API 没有重试。网络抖动或 API 限流时直接降级到模板摘要，丢失了 LLM 生成质量。

**方案**：

新增 `app/llm_retry.py`：

```python
import time
from functools import wraps

def retry_on_transient(max_retries=2, base_delay=1.0, backoff=2.0):
    """对 transient 错误（网络超时、429、5xx）重试，对 permanent 错误（400、401）不重试"""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_exc = None
            for attempt in range(max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except RuntimeError as exc:
                    last_exc = exc
                    if not is_transient(str(exc)) or attempt == max_retries:
                        raise
                    time.sleep(base_delay * (backoff ** attempt))
            raise last_exc
        return wrapper
    return decorator

def is_transient(error_msg: str) -> bool:
    transient_markers = ["timeout", "429", "500", "502", "503", "rate limit", "connection"]
    return any(marker in error_msg.lower() for marker in transient_markers)
```

在 `summarize_with_llm()` 的两个 API 调用处加 `@retry_on_transient()` 装饰器。

**影响文件**：

- 新增：`app/llm_retry.py`
- 修改：`app/summarize.py`
- 新增：`tests/test_llm_retry.py`

**验收标准**：

- 429/5xx 错误自动重试 2 次，间隔 1s/2s。
- 400/401 错误不重试，直接降级。
- 重试成功后正常返回 LLM 摘要。

def fetch_detail_text(session, url, max_chars=2000, timeout=10) -> str | None:
    """抓取详情页正文，失败返回 None（不阻断）"""
    try:
        text = fetch_html(session, url, timeout=timeout)
        soup = BeautifulSoup(text, "html.parser")

        content = soup.select_one(".content, #content, .article, article, .v_news_content")
        if content:
            return content.get_text(separator="\n", strip=True)[:max_chars]
        return None
    except Exception:
        return None

### 3.2 跨源去重

**背景**：不同源可能报道同一事件（如 RSS 报了一条半导体新闻，网页采集也抓到了同一条）。当前按 `(source_type, hash)` 去重，不同 source_type 之间不会去重。

**方案**：

做成 `enricher` 插件，在 collector 阶段完成后对 `ctx.state["web_items"]` 做标题相似度去重：

```python
def dedupe_cross_source(items, threshold=0.6):
    """基于标题 Jaccard 相似度的跨源去重"""
    result = []
    seen_titles = []
    for item in items:
        title_tokens = set(jieba.cut(item["title"]))  # 或简单字符 n-gram
        if any(jaccard(title_tokens, st) >= threshold for st in seen_titles):
            continue
        seen_titles.append(title_tokens)
        result.append(item)
    return result
```

为避免引入 `jieba` 依赖，可以用字符级 bigram Jaccard：

```python
def char_bigrams(text: str) -> set[str]:
    return {text[i:i+2] for i in range(len(text)-1)}

def jaccard(a: set, b: set) -> float:
    return len(a & b) / len(a | b) if a | b else 0.0
```

**影响文件**：

- 新增：`app/dedup.py`
- 修改：`app/plugins/builtins.py`（新增 `CrossSourceDedupePlugin`）
- 修改：`config/plugins.yml`（新增 `cross_source_dedupe`，默认关闭）
- 新增：`tests/test_dedup.py`

### 3.4 schema version 机制

**背景**：`db.py` 的 `ensure_migrations()` 用 `ALTER TABLE ADD COLUMN` 做迁移，每次连接都执行。随着表结构演进，这种方式不可维护。

**方案**：

```python
SCHEMA_VERSION = 2  # 每次 schema 变更递增

def ensure_migrations(conn):
    current = conn.execute("PRAGMA user_version").fetchone()[0]
    if current < 1:
        migrate_v1(conn)
    if current < 2:
        migrate_v2(conn)
    conn.execute(f"PRAGMA user_version = {SCHEMA_VERSION}")

def migrate_v1(conn):
    """items 表增加 first_seen_at, last_seen_at, last_notified_slot"""
    columns = {row["name"] for row in conn.execute("PRAGMA table_info(items)").fetchall()}
    for name in ["first_seen_at", "last_seen_at", "last_notified_slot"]:
        if name not in columns:
            conn.execute(f"ALTER TABLE items ADD COLUMN {name} TEXT")
    # ... backfill ...

def migrate_v2(conn):
    """新增 deadline_events 表"""
    conn.execute("""CREATE TABLE IF NOT EXISTS deadline_events (...)""")
```

实现注意：

- `connect()` 当前会先执行完整 `SCHEMA`，再跑 `ensure_migrations()`；改造时要确保新库、旧库、半迁移库都可重复执行。
- 迁移函数必须保持幂等，不能只依赖 `user_version`，仍要用 `PRAGMA table_info` 防止旧库字段状态不一致。
- 表创建可以继续放在 `SCHEMA`，但 schema 演进必须有对应 `migrate_vN()` 和 `PRAGMA user_version` 测试。
- 不要在第一个版本迁移里一次性塞入所有未来字段；后续 `deadline_events`、`content_full` 分别进入明确版本。

**影响文件**：

- 修改：`app/db.py`
- 修改：`tests/test_db.py`（覆盖新库、旧 schema、重复执行迁移）

## 风险控制

| 风险 | 控制方式 |
|---|---|
| BeautifulSoup 解析结果与旧正则不一致 | 对现有 4 个西交源做 A/B 对比测试，结果条目数和标题一致后才切换 |
| Jinja2 模板渲染 HTML 与旧 f-string 不一致 | 用 difftext 对比生成的 HTML，确保结构一致 |
| 采集并发导致 SQLite 写冲突 | 严禁跨线程共享同一个 sqlite connection；并发只做网络抓取，DB 写入和 `ctx.state` 合并回主线程 |
| 截止日期误抽取 | 置信度阈值 < 0.5 的不自动置顶，只在详情中展示 |
| 飞书增量推送漏推 | 保留 `--force-send` 全量推送能力；增量窗口可配置 |
| LLM 重试导致超时 | 最多重试 2 次，总等待不超过 5s；超时后仍降级到模板 |
