# 功能拓展计划：复刻飞书生态能力到个性日报助手

制定日期：2026-05-25

## 1. 结论

当前项目已经具备稳定基础：GitHub/RSS/网页采集、SQLite 去重、HTML 归档、邮件发送、飞书 Markdown 推送、来源健康记录和 delivery slot 幂等。

参考 `lark-cli`、飞书官方 OpenAPI、`larksuite/oapi-sdk-python` 以及 GitHub 上常见飞书 SDK/机器人项目后，最值得复刻的不是“重做一个完整飞书机器人平台”，而是把飞书变成当前日报的操作入口和沉淀层：

- 短期复刻：更好的飞书消息、分级推送、收藏/忽略/查询命令。
- 中期复刻：飞书 Base 看板、飞书任务、飞书文档周报。
- 长期复刻：事件订阅机器人、卡片交互、飞书 HTML 应用。

SQLite 仍作为主数据库；飞书只做入口、提醒和协作层。这样不会破坏当前低资源服务器约束。

## 2. 参考来源

- `lark-cli`：官方飞书/Lark CLI，覆盖 IM、Docs、Base、Sheets、Calendar、Mail、Tasks、Meetings、Markdown、Apps 等域，并提供 shortcut、API command、raw API 三层命令体系。参考：<https://github.com/larksuite/cli>
- `larksuite/oapi-sdk-python`：官方 Python 服务端 SDK，封装 token、签名、加解密、事件分发、类型化请求响应，并提供 Channel 模块用于消息事件、回复、媒体和卡片交互。参考：<https://github.com/larksuite/oapi-sdk-python>
- `chyroc/lark`：Go 飞书 SDK，强调全 OpenAPI 与事件回调覆盖、mock 测试、自建应用与 ISV、UserAccessToken、代码生成。参考：<https://github.com/chyroc/lark>
- 飞书 OpenAPI 文档：消息发送、文档、多维表格、任务等服务端 API。参考：<https://open.feishu.cn/document/>
- 本地 `lark-cli --help`：当前机器已安装 CLI，支持 `im`、`base`、`task`、`docs`、`drive`、`event`、`apps`、`markdown` 等命令。

| HTML 应用 | `lark-cli apps` | 把本地 `public/archive` 或小看板发布为飞书应用 | P3 | 等数据结构稳定后再做 |
| 日历联动 | `lark-cli calendar` | 截止日期、考试、报名提醒进入日历 | P3 | 只处理强时点事项，避免日历噪声 |

## 4. 与当前仓库的适配判断

### 已经具备的基础

- `app/lark_sender.py` 已经通过 `lark-cli im +messages-send` 发送 Markdown，并带 `idempotency-key`。
- `app/run_daily.py` 已有 `--collect-only`、`--send-only`、`--delivery-slot`，适合分离采集与推送。
- `app/db.py` 已有 `lark_logs`、`source_runs`、`source_health`、`items.first_seen_at`、`items.last_seen_at`、`items.last_notified_slot`。
- `app/render.py` 已有 HTML 归档和来源健康展示。
- `config/profile.yml` 已有偏好配置，可扩展为飞书频道、强提醒、忽略规则。

### 需要补齐的基础

- `last_notified_slot` 目前写入字段存在，但还没有按条目更新通知状态。
- 缺少 `saved_items`、`ignored_rules`、`user_actions`、`deadline_events`、`lark_sync_map`。
- 飞书消息没有分块、摘要降级、卡片模板、Base/任务同步。
- 缺少统一 `lark-cli` 包装层，当前 `lark_sender.py` 直接拼命令，后续会重复。

## 5. 功能分期

### 阶段 1：飞书消息体验增强

目标：让飞书推送比邮件更适合手机扫读。

任务：

- 新增 `app/lark_cli.py`，封装 `lark-cli` 调用、JSON 解析、超时、错误截断、dry-run。
- 重构 `app/lark_sender.py`，保留现有 env：`LARK_SEND`、`LARK_AS`、`LARK_USER_ID`、`LARK_CHAT_ID`。
- 新增 `render_lark_digest()`：
  - 今日必须处理
  - 稍后阅读
  - 开源项目
  - 来源异常
  - 归档路径
- 增加消息长度控制：超过阈值时只发摘要和归档路径。
- 增加 `--lark-only-important`：只推 score 高、今日新增或命中强提醒关键词的条目。
- 发送成功后更新 `items.last_notified_slot`。

验收：

- `python -m app.run_daily --sample --skip-mail --force-send` 仍可用。
- 飞书消息不因内容过长失败。
- 同一 slot 不重复推送。
- 推送失败不阻断邮件和 HTML 归档。

### 阶段 2：本地收藏、忽略、查询

目标：先在 SQLite 内建立反馈闭环，再接飞书事件。

新增表：

```sql
CREATE TABLE saved_items (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  item_id INTEGER NOT NULL,
  note TEXT,
  source TEXT NOT NULL DEFAULT 'local',
  created_at TEXT NOT NULL,
  UNIQUE(item_id)
);

CREATE TABLE ignored_rules (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  rule_type TEXT NOT NULL,
  pattern TEXT NOT NULL,
  enabled INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL,
  UNIQUE(rule_type, pattern)
);

CREATE TABLE user_actions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  action_type TEXT NOT NULL,
  item_id INTEGER,
  payload_json TEXT NOT NULL DEFAULT '{}',
  source TEXT NOT NULL,
  created_at TEXT NOT NULL
);
```

新增命令：

```bash
python -m app.brief_cli today
python -m app.brief_cli recent --days 7
python -m app.brief_cli search "EDA"
python -m app.brief_cli save --url "https://..."
python -m app.brief_cli ignore --keyword "娱乐"
python -m app.brief_cli health
```

验收：

- 可在本地查到最近日报条目。
- 收藏和忽略规则影响下一次评分/输出。
- 这些命令后续可直接挂到飞书机器人命令。

### 阶段 3：飞书 Base 看板同步

目标：把重要条目和来源状态同步到飞书多维表格，方便筛选、复盘和手工标记。

推荐 Base 表：

| 表 | 字段 |
| --- | --- |
| Items | 标题、URL、来源、分类、分数、状态、首次发现、最近发现、归档日期、本地 item_id |
| Sources | 来源名、类型、URL、最近状态、成功次数、失败次数、最近错误、耗时 |
| Actions | 动作、条目、备注、来源、创建时间 |
| Deadlines | 标题、截止时间、地点、状态、关联条目、飞书 task_id |

环境变量：

```text
LARK_BASE_APP_TOKEN=
LARK_BASE_TABLE_ITEMS=
LARK_BASE_TABLE_SOURCES=
LARK_BASE_TABLE_ACTIONS=
LARK_BASE_TABLE_DEADLINES=
```

实现：

- 新增 `app/lark_base_sync.py`。
- 用 `lark-cli base +record-upsert` 或 `+record-batch-create` 同步。
- 新增 `lark_sync_map(local_type, local_id, remote_type, remote_id, updated_at)`，避免重复创建。
- 只同步高价值条目、收藏条目、来源健康，不把所有原始数据倾倒到 Base。

验收：

- Base 里能看到今日重要条目和来源异常。
- 重复运行不会重复创建同一记录。
- Base 失败只记日志，不影响本地日报。

### 阶段 4：截止日期抽取与飞书任务

目标：把真正需要行动的通知转成任务。

任务：

- 新增 `deadline_events` 表，保存标题、截止时间、地点、置信度、关联 item。
- 从校园通知标题/摘要/可选详情页中抽取：
  - 报名截止
  - 申报截止
  - 考试时间
  - 答辩时间
  - 确认时间
- 新增 `app/lark_task_sync.py`：
  - 命中强提醒且有时间点时创建飞书任务。
  - 保存 `task_id`。
  - 支持任务完成状态回写，后续可做事件订阅。

验收：

- 只有高置信度事项会建任务。
- 同一事项不会重复建任务。
- 任务标题含来源和日期，备注含原文链接和归档链接。

### 阶段 5：飞书文档周报

目标：每周自动沉淀一份可回看的文档。

任务：

- 新增 `app/weekly_report.py`：
  - 本周必须处理事项
  - 本周收藏
  - 本周高价值 GitHub 项目
  - 来源健康异常
  - 下周关注建议
- 先写本地 Markdown/HTML，再用 `lark-cli docs` 或 `lark-cli markdown` 创建飞书文档。
- 支持 `--week YYYY-WW`、`--skip-lark-doc`。

验收：

- 周报可本地生成。
- 飞书文档创建失败时不影响本地归档。
- 文档链接写回 `digests` 或单独 `reports` 表。

### 阶段 6：飞书事件机器人

目标：通过飞书消息直接操作日报助手。

指令设计：

```text
/日报 今天
/日报 最近
/日报 搜索 关键词
/日报 来源
/日报 健康
/日报 收藏 链接或序号
/日报 忽略 关键词
/日报 任务 链接或序号
```

实现路线：

- 第一版用 `larksuite/oapi-sdk-python` 的 Channel 模块，独立运行 `python -m app.lark_bot`。
- 只读命令先上线，写命令后上线。
- 所有命令都写入 `user_actions`。
- 机器人服务和 `run_daily.py` 解耦，避免常驻服务影响定时日报。

验收：

- 飞书里能查询今天/最近/来源健康。
- 收藏、忽略、转任务能改变 SQLite 状态。
- 机器人挂了不影响定时日报。

### 阶段 7：卡片交互与飞书应用

目标：提升操作效率，但不作为前期必需项。

可做功能：

- 交互卡片：每条重要通知带“收藏”“忽略”“转任务”“打开归档”。
- 飞书应用：把 `public/archive` 或一个轻量查询页发布为飞书应用。
- 云空间归档：周报和关键 HTML 上传到飞书 Drive。

推迟原因：

- 卡片回调需要事件服务和更多权限。
- HTML 应用会引入部署/权限/分享范围管理。
- 当前最核心瓶颈仍是数据质量、分级推送和反馈闭环。

## 6. 推荐近期执行顺序

1. 新增 `app/lark_cli.py`，统一调用 `lark-cli`。
2. 做飞书消息分块、长度控制和 `--lark-only-important`。
3. 发送成功后更新 `items.last_notified_slot`。
4. 新增 `saved_items`、`ignored_rules`、`user_actions`。
5. 新增本地 `brief_cli` 查询/收藏/忽略命令。
6. 抽取截止日期，新增 `deadline_events`。
7. 同步重要条目到 Base。
8. 高置信度截止事项创建飞书任务。
9. 生成飞书周报文档。
10. 最后再上事件机器人和交互卡片。

## 7. 不建议复刻的部分

- 不直接把项目改成全量飞书 SDK 服务。当前定时脚本架构更稳定，服务器资源也更合适。
- 不用 Base 替代 SQLite。Base 适合看板和协作，不适合做本地幂等、抓取状态和离线回退的唯一事实源。
- 不一开始做实时机器人。先用本地命令和同步任务跑通数据模型，再接事件。
- 不把所有 HTML 归档、原始网页、附件都上传飞书。只上传摘要、周报和用户收藏内容。
- 不先做飞书应用页面。当前最有价值的是消息入口、任务和 Base，而不是新的前端页面。

## 9. 权限和安全边界

- 继续使用现有 `lark-cli` 授权，不在 `.env` 中新增 app secret，除非后续上 SDK 事件机器人。
- Base、Task、Docs 权限按阶段申请，不一次性开过大权限。
- 所有飞书远端 ID 只保存 token/id，不保存用户隐私内容。
- 日志不能打印 token、邮箱密码、完整授权响应。
- 所有远端同步必须幂等，失败可重试，不影响本地 SQLite。

## 10. 目标形态

最终体验：

- 早晨：飞书收到完整日报。
- 白天：只收到新增且高优先级的通知。
- 看到有价值条目：在飞书里收藏或转任务。
- 想回看：在飞书里搜索最近条目，或打开 Base 看板。
- 每周：自动生成飞书文档周报。
- 出问题：来源健康和飞书同步状态可见，日报主流程不被阻断。
