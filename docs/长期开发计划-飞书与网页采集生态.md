# 长期开发计划：飞书与网页采集生态

更新时间：2026-05-25

## 1. 长期目标

把当前项目从“定时日报脚本”逐步发展成一个轻量、可维护、可交互的个人情报助手：

- 网页采集负责持续发现公开信息：西交官网、学院官网、竞赛通知、技术博客、开源项目和半导体/EDA 信息源。
- RSS/Atom 负责接入通用新闻和产业资讯：半导体、EDA、电子工程、技术趋势。
- 飞书负责日常入口：推送、检索、收藏、提醒、配置和任务跟踪。
- SQLite 继续做本地事实库，避免服务器负担过重。
- LLM 只做筛选、摘要、归类、行动建议，不承担长期存储和调度。
- 所有功能都必须保留现有 SMTP、Lark、cc-switch、GitHub 授权方式，不做破坏性改造。

## 2. 设计原则

- 轻量优先：2 核 2 GiB 服务器不跑本地大模型、不常驻浏览器、不做大规模爬虫。
- 来源透明：每条日报必须有原链接、来源名、抓取时间和评分理由。
- 单源失败不阻断：某个官网挂了，只记录来源状态，不影响其它内容发送。
- 配置驱动：新增网页源优先改 `config/sources.yml`，不要为每个站写死流程。
- 飞书渐进增强：先做推送和指令，再做 Base/任务/文档联动。
- 可回滚：新增功能默认关闭或低风险启用，不改现有授权变量。
- 健康状态持久化：来源成功、失败、耗时和条目数必须写入 SQLite，`send-only` 也能看到来源状态。
- 新增判断先行：`first_seen_at`、`last_seen_at`、`last_notified_slot` 是分级推送和“今日新增”的数据基础。

## 3. 子 agent 审阅后的关键调整

审阅结论：路线整体正确，但第 1 批必须先补数据基础，否则飞书分级推送、查询和收藏都会建立在不稳定状态上。

已采纳调整：

- `source_runs` / `source_health` 提前到第 1 批，并要求跨 collect/send 可用。
- `items.first_seen_at` / `items.last_seen_at` / `items.last_notified_slot` 提前到第 1 批，为“今日新增”和补推做准备。
- 飞书失败不能阻断 SMTP、归档和普通日报，后续 Base/任务/文档联动也必须独立降级。
- 飞书消息长度控制提前到第 2 批之前，避免来源扩展后手机端不可读。
- 飞书指令拆成两阶段：先只读查询，再收藏/忽略/加源等写操作。
- “加源 URL 名称”推迟到配置校验和抓取预览完成后，避免任意网页抓取带来噪声。
- 通用新闻/RSS 和产业资讯不再放到最后，先接入少量已验证可用的高质量源。

## 4. 阶段路线

### 阶段 A：稳定采集与输出

周期：1 到 2 周。

目标：让当前西交官网和 GitHub 日报稳定运行。

任务：

- 给每个 `webpages` 来源增加健康状态：成功次数、失败次数、最近错误、最近成功时间。
- 给 RSS/Atom 来源同样增加健康状态。
- 给 `items` 增加首次发现、最近发现、最近通知 slot 字段。
- 在 HTML 日报中显示“来源健康”摘要。
- 给网页采集增加更明确的超时、重试和跳过策略。
- 接入少量通用新闻/RSS 产业源，先控制数量，观察稳定性。
- 增加正文详情抓取的可选开关，只对重要通知抓详情页。
- 抽取通知中的截止日期、报名时间、考试时间、地点等行动字段。
- 增加 `--dry-run`，只抓取和渲染，不写发送日志。

验收：

- `python -m unittest discover -v` 通过。
- 官网源失败时仍能生成日报。
- RSS 源失败时仍能生成日报。
- 日报中能看到来源状态和关键行动字段。

### 阶段 B：飞书消息体验升级

周期：2 到 4 周。

目标：让飞书不只是收日报，而是个人情报入口。

任务：

- 优化飞书 Markdown 输出，分成“今日必须处理”“稍后阅读”“来源异常”。
- 支持飞书推送分级：
  - 早晨：完整日报。
  - 中午/下午：只推新增高优先级通知。
  - 晚上：今日未处理事项回顾。
- 给每条飞书消息附本地归档路径和原始链接。
- 增加消息幂等键，继续避免重复发送。
- 增加 `--lark-only-important`，只发送高分条目。
- 增加飞书消息长度控制，超长内容只发摘要和归档链接。

验收：

- 飞书消息比邮件更适合手机快速扫读。
- 同一 delivery slot 不重复发送。
- 低优先级内容不会频繁打扰。

### 阶段 C：飞书交互指令

周期：1 到 2 个月。

目标：能在飞书里控制日报助手。

第一阶段只读指令：

```text
/日报 今天
/日报 最近
/日报 来源
/日报 搜索 关键词
/日报 健康
```

第二阶段写入指令：

```text
/日报 收藏 链接
/日报 忽略 关键词
/日报 加源 URL 名称
```

实现路线：

- 第一版不做复杂机器人服务，先用 `lark-cli` 和定时脚本生成可读结果。
- 后续如果需要实时交互，再用飞书事件订阅或 webhook。
- 指令结果仍写入 SQLite，保证飞书不是唯一数据源。
- 加源命令必须等配置校验、域名 allowlist 和抓取预览完成后再开放。

数据表建议：

```text
user_actions
saved_items
ignored_rules
source_health
```

验收：

- 能通过飞书查看最近日报。
- 能收藏某条通知或项目。
- 能查询来源健康。
- 交互失败不会影响定时日报。

### 阶段 D：网页采集生态扩展

周期：1 到 3 个月。

目标：从几个固定官网扩展成可维护的信息源系统。

优先来源：

- 西交教务处、本科生院、研究生院、学工部、就业创业、创新港、学院官网。
- 竞赛官网：集创赛、研究生电子设计竞赛、开源操作系统大赛等。
- 通用新闻/RSS：IEEE Spectrum、HN、arXiv 关键词等。
- 产业源：Semiconductor Engineering、EDN、EDA 厂商、晶圆厂、IP、先进封装、开源 PDK。

功能：

- `source_type` 扩展：`school_notice`、`competition`、`tech_blog`、`github_release`、`industry_news`。
- RSS/Atom 统一解析成 `rss_entry` / `industry_news` / `tech_news`。
- 每个来源定义抓取频率和重要度。
- 对正文进行正文抽取和附件链接提取。
- 对 PDF/Doc 附件只保存链接和标题，不默认下载，避免磁盘压力。
- 对列表页变更做 hash 比较，减少重复处理。

验收：

- 新增 10 个来源只需要改配置或少量 parser。
- 单次采集在普通服务器上稳定完成。
- 数据库和归档增长可控。

### 阶段 E：飞书 Base / 文档联动

周期：2 到 4 个月。

目标：把重要条目沉淀成可管理的个人知识库。

可选方向：

- 飞书 Base 保存来源配置、收藏条目、待办状态。
- 飞书任务创建“需要处理”的通知，例如报名、申报、确认、答辩。
- 飞书文档自动生成周报或月报。
- 飞书多维表格作为可视化看板：来源、分类、状态、优先级、截止日期。

边界：

- Base/任务/文档只做增强，不替代 SQLite。
- 飞书 API 失败时，本地日报仍可正常运行。
- 不把密钥、个人授权信息写入文档或数据库明文展示。

验收：

- 重要通知能一键沉淀为飞书任务或 Base 记录。
- 周报能从本地 SQLite 或 Base 生成。
- 飞书联动失败有日志和降级方案。

### 阶段 F：个性化智能层

周期：长期。

目标：日报越来越懂个人偏好，但规则可解释。

功能：

- 从 `config/profile.yml` 管理兴趣方向、降权关键词、强提醒关键词。
- 对条目生成评分解释：为什么推荐、适合做什么、是否需要行动。
- 支持“已读/收藏/忽略”反馈反向调整评分。
- 按学习、科研、竞赛、开源工具、校园事务分频道输出。
- 生成每周复盘：错过的截止日期、收藏未读、连续高频来源。

验收：

- 用户能看懂推荐理由。
- 忽略规则生效。
- 周报能指出真正需要处理的事项，而不是堆列表。

## 5. 推荐开发顺序

1. 先做来源健康持久化、首次/最近发现字段、RSS 基础采集。
2. 再做截止日期抽取和来源配置校验。
3. 再做飞书消息分级推送和长度控制。
4. 然后做飞书只读查询。
5. 再做收藏、忽略、任务/Base/文档联动。

这样排序原因：

- 前两步补齐后续推送和查询的数据基础。
- 飞书交互必须建立在稳定数据之上。
- Base/任务联动涉及更多授权和失败场景，应等本地闭环稳定后做。

## 7. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 官网结构变化 | parser 测试 + 来源健康状态 |
| RSS 源超时或改版 | 单源超时 + 健康表 + 不阻断日报 |
| 飞书 API 限流 | 分级推送 + 幂等键 + 失败降级 |
| 服务器压力 | requests 优先、限制来源数量、限制详情页抓取 |
| SQLite 增长 | 归档清理、旧条目压缩、附件不默认下载 |
| 推送打扰 | 高优先级模式、时间段控制、关键词降权 |
| 授权破坏 | 不改现有 env 名称，不改 `lark_sender.py` 基础发送方式 |

## 8. 长期完成形态

最终形态不是一个复杂平台，而是一个稳定的个人工作流：

- 早晨收到飞书日报，知道今天要处理什么。
- 白天只收到真正紧急或高价值的新增提醒。
- 任何通知和项目都能在飞书里搜索、收藏、转成任务。
- 通用新闻、RSS 和产业资讯能和校园通知同屏比较优先级。
- 每周自动生成复盘，沉淀到飞书文档或本地归档。
- 新增来源成本低，失败可见，服务器压力可控。
